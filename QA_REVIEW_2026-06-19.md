## Findings

### 1. Invalid sync codes are accepted as valid collections
Severity: high

`/synccollection` trusts `sprites.decode(code)` and only rejects `ValueError`s, but the decoder uses lenient base64 decoding and never validates the decoded payload length or shape. As a result, obvious garbage like `!!!!` and `abc` is accepted and stored as a collection instead of being rejected.

Confirmed locally:
- `!!!!` decoded successfully
- `abc` decoded successfully
- both can lead to silent junk collection state

Relevant code:
- [spritebot/sprites.py](/Users/bryan/sprite-trade-stop/spritebot/sprites.py:52)
- [spritebot/cogs/collection_sync.py](/Users/bryan/sprite-trade-stop/spritebot/cogs/collection_sync.py:48)

Why this matters:
- users can accidentally overwrite their collection with bad data
- fabricated or malformed sync strings are treated as legitimate
- this weakens trust in `/synccollection`, `/spriteinfo`, `/guildprogress`, and `/spritematch`

Suggested fix:
- use strict base64 validation
- reject payloads that do not match the expected encoded length for `SHARE_ORDER`
- optionally add a version byte or checksum to future sync codes

### 2. A trader can only ever receive one active vouch from the same person
Severity: medium-high

The DB rule in `already_vouched` prevents a second active vouch from the same `voucher_id -> target_id` pair forever unless the first vouch is removed. In a real repeat-trading community, this blocks legitimate repeat trust accrual between regular trading partners.

Relevant code:
- [spritebot/db.py](/Users/bryan/sprite-trade-stop/spritebot/db.py:178)
- [spritebot/cogs/vouch.py](/Users/bryan/sprite-trade-stop/spritebot/cogs/vouch.py:42)

Why this matters:
- repeat successful trades stop contributing to trust progression
- leaderboard and flair progression flatten for active repeat traders
- product behavior may feel unfair or confusing after the first trade

This may be intentional anti-farming, but in its current form it is too blunt.

Suggested fix:
- tie vouches to completed trade IDs
- or allow repeated vouches after distinct completed trades
- or allow repeated vouches after a cooldown window plus anti-abuse checks

### 3. `/whoneeds` and `/match` produce misleading trade targets
Severity: medium

The role-based collection flow treats every non-bot member without a sprite role as someone who “needs” that sprite. `/match` then pairs holders and seekers by list order, not by actual reciprocal fit or expressed intent.

Relevant code:
- [spritebot/cogs/collection.py](/Users/bryan/sprite-trade-stop/spritebot/cogs/collection.py:63)
- [spritebot/cogs/collection.py](/Users/bryan/sprite-trade-stop/spritebot/cogs/collection.py:86)

Why this matters:
- users who never completed onboarding appear to need everything
- “matches” can look smarter than they really are
- the command can send traders toward low-quality or false-positive targets

The repo already acknowledges this in [ISSUES.md](/Users/bryan/sprite-trade-stop/ISSUES.md:9), so I’d treat this as a real known product limitation, not just a nitpick.

Suggested fix:
- scope seekers to users with relevant onboarding or synced collection state
- rename `/match` if it remains heuristic
- or upgrade matching to use synced collection overlap instead of arbitrary pairing

### 4. The bot now has two collection truth systems that can disagree
Severity: medium

The bot currently uses:
- onboarding roles for `/whohas`, `/whoneeds`, `/match`, `#sprite-list`, `#gold-zp-list`
- synced DB collections for `/holders`, `/spritematch`, `/spriteinfo`, `/guildprogress`, `/mycollection`, `/missing`, `/spriteset`

Relevant code:
- [spritebot/cogs/collection.py](/Users/bryan/sprite-trade-stop/spritebot/cogs/collection.py:1)
- [spritebot/cogs/collection_sync.py](/Users/bryan/sprite-trade-stop/spritebot/cogs/collection_sync.py:1)

Why this matters:
- the same user can appear differently depending on the command
- there is no reconciliation strategy
- there is no strong user-facing explanation of which source is authoritative for which command

This is not a crash bug, but it is the biggest product-clarity risk in the current architecture.

Suggested fix:
- explicitly define “roles are public trade intent, sync is private/full collection truth”
- surface that distinction in help text and embeds
- or move toward one canonical collection model and derive the other presentation from it

### 5. The “daily leaderboard” is not actually clock-time daily and will create channel noise
Severity: medium-low

The leaderboard loop posts every 24 hours from process start and always sends a new message. Restarting the bot shifts the post time, and long-lived servers will accumulate repeated leaderboard posts instead of one maintained message.

Relevant code:
- [spritebot/cogs/admin.py](/Users/bryan/sprite-trade-stop/spritebot/cogs/admin.py:130)
- [spritebot/cogs/admin.py](/Users/bryan/sprite-trade-stop/spritebot/cogs/admin.py:143)

This is also already acknowledged in [ISSUES.md](/Users/bryan/sprite-trade-stop/ISSUES.md:15).

Suggested fix:
- switch to a fixed-time schedule
- optionally store and edit a known leaderboard message instead of always sending a new one

## Product Read

Right now the bot is best described as:

**a trading/vouch bot with collection features**

not yet:

**a fully coherent Sprite tracker platform**

That is fine if the server’s core goal is trust + trading.

If the goal is to become the best Fortnite Sprite tracker/community tool, the next step is not more random commands. The next step is making the collection model coherent and trustworthy:
- strict sync validation
- one clear truth model
- better matching quality
- smarter repeat-trade reputation rules

## Recommended Next Priorities

1. Fix sync-code validation first.
2. Decide the canonical relationship between onboarding roles and synced collections.
3. Rework vouch rules so repeat legitimate trades can count safely.
4. Upgrade `/match` and `/whoneeds` so they stop over-reporting false targets.
5. Add tests for the newer collection-sync feature wave.

## QA Verdict

Overall verdict: **promising and real, but not done hardening yet**

This is no longer a toy bot. Claude has built a real feature set, and the recent additions were worthwhile.

The most important remaining issues are not “style” problems:
- trust in imported collection data
- coherence between the two collection systems
- accuracy of match/need suggestions
- fairness of long-term reputation growth

If those are cleaned up, this moves from “good Discord utility bot” toward “strong niche community product.”
